<objective>
Create mobile-optimized API endpoints that return Shot Card bundles paginated shot-by-shot with progressive loading support.

Purpose: Mobile devices need a dedicated API that bundles visual + audio + narrative data per shot, with smaller page sizes and cursor-based pagination optimized for card-by-card consumption. The existing /api/v1/shot-cards endpoint returns raw ShotCardResponse objects; the mobile endpoint transforms them into mobile-friendly bundles.

Output: Two new endpoints (GET /api/v1/mobile/cards, GET /api/v1/mobile/cards/{id}/audio) plus new Pydantic schemas.
</objective>

Create `app/api/v1/mobile.py` with a FastAPI APIRouter (prefix="/api/v1/mobile", tags=["mobile"]).

**Helper function** `_shot_card_to_bundle(shot_card: ShotCard) -> MobileShotCardBundle`:
Extract fields from the ShotCard ORM object's JSONB columns into the flat MobileShotCardBundle:
- `nc = shot_card.narrative_context or {}`
- `scene = nc.get('scene', '')`, `shot_number = nc.get('shot_number', 0)`, etc.
- `vb = shot_card.visual_bundle or {}`
- `first_frame_url` from `vb -> keyframes -> first -> url` (safely nested dict access using chained .get())
- `last_frame_url` from `vb -> keyframes -> last -> url`
- `video_url` from `vb -> video_clip -> url`
- Same pattern for `ab = shot_card.audio_bundle or {}`

**Endpoint 1: GET /api/v1/mobile/cards**
- Query params: `project_id: str | None`, `cursor: int | None`, `limit: int = Query(10, ge=1, le=50)`
- Default limit=10 (mobile-friendly page size, smaller than desktop's 30)
- Filters: only `audit_status == AuditStatus.AWAITING_AUDIT` (mobile only shows cards needing review)
- Optional filter by `project_id`
- Cursor-based pagination: `ShotCard.id > cursor` ordered by `ShotCard.id asc`
- Returns `ApiResponse[PaginatedResponse[MobileShotCardBundle]]`
- Use the same pagination pattern as shot_cards.py: fetch limit+1, check has_more, slice to limit

**Endpoint 2: GET /api/v1/mobile/cards/{shot_card_id}/audio**
- Returns `ApiResponse[MobileAudioBundle]` for a single shot card
- Purpose: async audio loading -- mobile client calls this after initial visual load
- Returns 404 if shot card not found
- Extracts only audio_bundle fields from the ShotCard

**Endpoint 3: POST /api/v1/mobile/cards/{shot_card_id}/swipe-decision**
- Request body: `action: Literal["approve", "reject"]`, `reason: str | None = None`
- Reject requires reason (validate: if action=="reject" and not reason, return 422)
- Same logic as existing approve/reject endpoints: check audit_status == AWAITING_AUDIT, transition to APPROVED/REJECTED
- Returns `ApiResponse[MobileShotCardBundle]` (updated bundle)
- Requires JWT auth via `Depends(get_current_client)`
