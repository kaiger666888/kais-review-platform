# Phase 3 — UI Design Contract

> Visual and interaction contract for the Review Frontend phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (not React/Next.js — Tailwind v4 CDN utility classes) |
| Preset | not applicable |
| Component library | none (hand-crafted Jinja2 partials with Tailwind utilities) |
| Icon library | Heroicons SVG inline (outline style, 24px) — zero dependency, copy-paste into templates |
| Font | System font stack: `ui-sans-serif, system-ui, -apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif` |

**Rationale:** This is a FastAPI + HTMX project with zero build step. There is no npm, no React, no component framework. shadcn is not applicable. All styling is Tailwind v4 CDN utility classes applied directly in Jinja2 HTML templates. Icon SVGs are inlined for zero network requests on a LAN deployment with no internet.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Tailwind Class | Value | Usage |
|-------|----------------|-------|-------|
| xs | `p-1` / `gap-1` | 4px | Icon gaps, inline padding, badge spacing |
| sm | `p-2` / `gap-2` | 8px | Compact element spacing, card inner padding |
| md | `p-4` / `gap-4` | 16px | Default element spacing, card padding |
| lg | `p-6` / `gap-6` | 24px | Section padding, content area margins |
| xl | `p-8` / `gap-8` | 32px | Layout gaps, header-to-content spacing |
| 2xl | `p-12` / `gap-12` | 48px | Major section breaks |
| 3xl | `p-16` / `gap-16` | 64px | Page-level spacing (rarely used on mobile) |

Exceptions:
- Bottom tab bar height: 56px (`h-14`) — standard mobile bottom nav height
- Touch target minimum: 44px (`min-h-[44px] min-w-[44px]`) — WCAG 2.5.5 compliance for mobile tap targets
- Review card action buttons: 48px height (`h-12`) — generous for one-tap approve/reject

|------|------|----------------|--------|-------------|
| Body | 14px | `text-sm` | 400 (regular) | 1.5 (`leading-relaxed`) |
| Label | 12px | `text-xs` | 600 (semibold) | 1.33 (`leading-normal`) |
| Heading | 18px | `text-lg` | 600 (semibold) | 1.33 (`leading-normal`) |
| Display | 24px | `text-2xl` | 600 (semibold) | 1.2 (`leading-tight`) |

**Usage rules:**
- Body (14px): review card descriptions, detail page content, toast messages
- Label (12px): status badges, source system tags, elapsed time, tab labels
- Heading (18px): review detail title, section headings, empty state headings
- Display (24px): dashboard header title, error page headings

## Interaction Contracts

### Dashboard (main view)

1. **Initial load**: GET `/` — server renders dashboard with pending reviews list
2. **Tab switch**: Click tab (pending/approved/rejected) — HTMX `hx-get="/reviews?status={status}"` targets review list container, swaps inner HTML
3. **SSE real-time**: `hx-sse="connect:/api/v1/events/stream"` listens for `review_status` events. On event, HTMX replaces `_review_list` partial for the active tab. New reviews that don't match the active tab increment the new reviews banner count.
4. **Load more**: Button at bottom of list — HTMX `hx-get="/reviews?status={status}&cursor={last_id}"` with `hx-swap="beforeend"` to append cards

### Review Card

- Tap card: HTMX `hx-get="/reviews/{id}/detail"` — renders `_review_detail` partial as full-screen overlay
- Shows: type badge (colored by disposition), source system label, elapsed time ("2m ago", "1h ago"), priority indicator dot (critical = red dot)

### Review Detail Overlay

- Full-screen slide-up overlay (Tailwind: fixed inset-0 with slide-up transition)
- Content: review metadata, content_ref preview (image thumbnail if available, text snippet otherwise), audit history summary
- Footer: two large buttons — "Approve" (green-600, full width left half) and "Reject" (red-600, full width right half)
- Approve: HTMX `hx-post="/reviews/{id}/approve"` — on success, close overlay, show toast "Review approved", update card state
- Reject: Opens reject confirmation dialog (Alpine.js overlay) — requires reason input (textarea, required) — then HTMX `hx-post="/reviews/{id}/reject"` with reason

### One-Time Token Deep Link

1. User opens `/t/{token}` on phone
2. Server validates token: if valid, exchanges for JWT (httpOnly cookie), redirects to `/reviews/{id}` with detail overlay open
3. If token invalid/expired: redirect to `/` with error toast

### Toast Notifications

- Fixed position top-center, slides down
- Auto-dismiss after 3 seconds (Alpine.js `x-init="setTimeout(() => remove(), 3000)"`)
- Types: success (green border), error (red border), info (blue border)

## Responsive Behavior

| Breakpoint | Layout |
|------------|--------|
| < 640px (mobile, primary) | Single column, full-width cards, bottom tab bar, full-screen overlays |
| 640-1024px (tablet) | Max-width container (640px) centered, cards in 2-column grid |
| > 1024px (desktop) | Max-width container (800px) centered, cards in 2-column grid, slightly larger typography |

**Primary target is < 640px mobile.** Tablet and desktop are graceful extensions, not separate layouts.

---

## Accessibility

- All buttons have visible text labels (no icon-only buttons)
- Color is never the sole indicator — status badges include text ("Pending", "Approved", "Rejected")
- Tap targets minimum 44px (WCAG 2.5.5)
- Focus visible styles: Tailwind `focus-visible:ring-2 focus-visible:ring-blue-600`
- Semantic HTML: `<nav>` for tab bar, `<main>` for content, `<article>` for review cards, `<dialog>` for confirmations
- ARIA: `role="tablist"` on tab bar, `role="tab"` on tabs, `aria-selected` on active tab

## Server-Side Rendering Patterns

| Pattern | Implementation |
|---------|---------------|
| Full page render | `render_template("pages/dashboard.html", reviews=..., active_tab=...)` |
| HTMX partial render | `render_template_block("pages/dashboard.html", "review_list", reviews=...)` via jinja2-fragments |
| HTMX response headers | `HX-Trigger: {"showToast": {"message": "Review approved", "type": "success"}}` for toast triggers |
| SSE event → HTML | Server emits event, HTMX replaces target container with server-rendered HTML fragment |
| JWT management | Server sets JWT in httpOnly cookie — browser never handles token directly |
