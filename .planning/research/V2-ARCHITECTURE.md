## 二、核心概念：Shot Card（分镜卡片）

### 2.1 审核原子单元

短剧管线的最小叙事单元不是"一张图"或"一段视频"，而是一个分镜（Shot）。所有共享同一 shot_id 的节点输出被聚合为一张 Shot Card：

| 捆绑内容 | 来源节点 | 审核价值 |
| --- | --- | --- |
| 首帧 / 尾帧 | 文生图 / 图生图 | 判定角色一致性、光影连续性 |
| 中间帧序列 / 视频片段 | 图生视频 (Wan2.2) | 判定运动合理性、运镜执行度 |
| 视觉提示词 | Prompt Node | 判定"词是否达意" |
| 音频提示词 | Audio Prompt Node | 判定 BGM 情绪与视觉匹配度 |
| 节点执行状态 | OpenClaw DAG | 显示哪些子任务已完成、哪些进行中 |

### 2.2 Shot Card 数据模型

```yaml
shot_card:
  shot_id: "scene02_shot07"
  project_id: "short_drama_alpha"
  narrative_context:
    scene: "雨夜告别"
    shot_number: 7
    emotion_curve: "孤独→释然"
    continuity_tags: ["角色A侧脸", "窗外雨景", "逆光轮廓"]

  visual_bundle:
    keyframes:
      first: { url: "...", hash: "7a8f9b...", node: "FLUX.1-dev" }
      last:  { url: "...", hash: "3c2d1e...", node: "FLUX.1-dev" }
    video_clip: { url: "...", duration: 4.2, node: "Wan2.2-T2V" }
    prompt: "东亚女性侧脸, 逆光轮廓, 窗外雨景, 缓慢推轨..."
    candidates:  # 同镜头多抽卡候选
      - candidate_id: "c01"
        keyframes: { first: "...", last: "..." }
        score: null  # 预留 AI 评分

  audio_bundle:
    bgm_prompt: "钢琴独奏, 小调, 72BPM, 情绪曲线: 上扬→收"
    sfx_prompt: "雨声渐强, 玻璃反光低频嗡鸣"
    status: "pending"  # pending / ready / failed

  audit_state:
    status: "awaiting_audit"  # awaiting_audit / approved / rejected / pending_audio
    routing_decision: "HUMAN"  # AUTO / HUMAN / AI_AUDIT / BLOCK
    min_audit_set: ["visual_bundle"]  # 最小可审集合
    blocking_reason: null

  provenance:
    workflow_version: "abc123..."
    policy_commit_sha: "7d8e9f..."
    execution_id: "exec_20260515_143022"
```

## 三、第一层：GitOps 版本控制平面

| 控制对象 | Git 管理内容 | 版本关联策略 |
| --- | --- | --- |
| 工作流定义 | OpenClaw 管线 JSON/YAML（节点拓扑、参数模板） | 每次变更产生 Git Tag，审计日志记录 workflow_version 哈希 |
| 审核策略 | 策略规则文件（YAML）：风险阈值、强制检查点列表、自动通过条件 | 策略变更需 PR 审批，审计记录引用 policy_commit_sha |
| AI 审计资产 | 评分模型配置、Prompt 模板、权重文件（存引用和哈希） | 模型版本与审计记录绑定，支持回滚到旧策略重新评估 |
| Shot Card 模板 | 分镜卡片结构定义、捆绑规则、最小可审集合配置 | 定义哪些节点输出必须聚合才能解锁审核 |
| 审计锚定 | 每日将审计日志的 Merkle Root 写入 Git | 实现审计日志的防篡改校验 |

关键原则：治理层的所有"决策逻辑"必须入 Git，运行时只读取特定 Commit；Shot Card 的捆绑规则也由 Git 版本化。

## 四、第二层：治理核心引擎层

### 4.1 策略引擎（Policy Engine）

- 输入：当前 Shot Card 的 narrative_context（风险标签、成本等级）、内容元数据、Git 中的策略文件
- 输出：路由决策（AUTO / HUMAN / AI_AUDIT / BLOCK）
- 设计：支持策略叠加——全局策略 + 项目策略 + 临时策略

### 4.2 Shot Card 聚合器（Shot Card Aggregator）

- 拓扑折叠：监听 OpenClaw 事件总线，将共享同一 shot_id 的节点输出渐进式聚合为 Shot Card
- 渐进式填充：视觉部分先完成就先显示，音频部分完成后自动附加
- 解锁规则：当 min_audit_set 中定义的最小可审集合（如视觉 bundle）全部就绪后，解锁审核按钮；音频未完成时可选择"通过但挂起音频"或"等待完整"

### 4.3 检查点管理器（Checkpoint Manager）

- 状态机：将 OpenClaw 的运行状态序列化为 RunState Snapshot，按 Shot Card 粒度存入高速 KV（Redis）
- 超时策略：人工审核默认 24h 超时转拒绝；AI 审核默认 5min 超时转人工
- 恢复机制：审核通过后注入 ResumeCommand，OpenClaw 从快照点继续执行下游节点

### 4.4 审批路由器（Approval Router）

- 动态分流：根据策略引擎输出，将 Shot Card 路由到三个出口之一
- 优先级队列：3090 高参渲染任务进入高优队列，抽卡预览进入低优队列
- 批量处理：同一场次的多个 Shot Card 可聚合为 Batch Approval

### 4.5 令牌服务（Token Service）

- 短活令牌：审批链接携带一次性 JWT（15min 有效）
- 能力令牌：审核通过后向 OpenClaw 执行层发放 Capability Token，未持令牌的高成本任务被执行层拒绝

### 4.6 审计记录器（Audit Recorder）

- 结构化 Schema
- 双写策略：实时写入时序数据库 + 异步归档不可变存储

### 4.7 事件总线（Event Bus）

- 推送通道：审核状态变更通过 SSE/WebSocket 推送到桌面端/移动端
- Webhook：向 Telegram 发送审核通知
- 渐进式填充事件：节点完成时实时更新 Shot Card 的局部状态

桌面端专属能力：

| 能力 | 实现 |
| --- | --- |
| 键盘快捷键 | Space 播放/暂停，→ 下一帧，Y 通过，N 拒绝，B 批量，Shift+Y 批量通过，J/K 切换 Shot，D Diff 对比，G 跳转 Git 策略，L 打开 OpenClaw 日志 |
| 双栏对比 | 首帧 vs 尾帧、当前候选 vs 历史版本、当前 vs 参考图 |
| 候选阵列 | 同一 Shot 的多抽卡结果以缩略图阵列展示，点击无缝切换 |
| 深度关联 | 点击 execution_id 查询日志；点击 Git Commit 直接跳转策略 Diff |
| 批量操作 | Ctrl/Shift 多选左栏 Shot，右栏一键批量决策 |
| 侧边栏常驻 | 可作为浏览器侧边栏或独立 PWA 窗口置顶，通知时自动弹出 |

### 5.2 人工审核网关——移动端（Mobile PWA）

卡片流布局：
- 纵向滑动：切分镜（Shot 07 → Shot 08），保持叙事连续性
- 横向滑动：切同一分镜的不同抽卡候选
- 卡片结构：首帧/尾帧始终可见（时空锚定），提示词/音频参数可折叠展开
- 手势操作：左滑通过、右滑拒绝、上滑查看详情、双指放大预览
- 上下文条：卡片顶部显示"场次2 - 第3个镜头 - 情绪：孤独→释然"
- 离线缓存：Service Worker 缓存最近 20 条 Shot Card

### 5.3 AI 审计预留窗口（AI Audit Extension Interface）

| 预留接口 | 职责 | 当前占位实现 |
| --- | --- | --- |
| 评分插件总线 | 接收 Shot Card，返回多维度评分向量（美学、一致性、合规、技术质量、音频匹配度） | 返回空向量，路由 fallback 到人工 |
| 模型注册中心 | 管理 AI 审计模型的版本、能力标签、置信度阈值 | 空注册表，标记 model_unavailable |
| 反馈闭环 | 人工审核结果回流，作为训练信号 | 数据入冷存储 |
| 渐进式控制器 | 根据 AI 评分历史准确率动态调整阈值 | 初始 threshold=1.0（全部人工） |
| A/B 测试接口 | 同一批 Shot Card 并行送 AI 和人工，对比一致性 | 预留数据格式 |
| 影子模式 | AI 评分持续运行但不影响决策 | 记录结果用于训练 |

关键设计：AI 窗口与人工网关共享同一套 Shot Card 数据结构，AI 的决策格式与人工完全一致。

## 八、AI 自动化审计渐进式演进路径

| 阶段 | 状态 | 治理层行为 |
| --- | --- | --- |
| Phase 0（当前） | 全人工 | AI 窗口返回空，Router 100% 导向人工网关 |
| Phase 1 | AI 辅助评分 | 评分插件运行，结果展示在 Shot Card 供参考，决策权仍在人 |
| Phase 2 | AI 预审 | 高置信度 Shot Card 自动通过，低置信度 + 边缘案例送人工 |
| Phase 3 | AI 主导 | 人工仅处理 AI 标记的"不确定"案例，比例 < 10% |
| Phase 4 | 全自动化 | 仅在检测到异常模式时触发人工复核 |

预留机制：
- A/B 测试接口：同一批 Shot Card 并行送 AI 和人工
- 影子模式：AI 评分持续运行但不影响决策
- 熔断机制：AI 与人工连续偏离超阈值时自动回退全人工

## 九、关键设计决策总结

1. **Shot Card 是审核原子**：将 OpenClaw 节点拓扑折叠为叙事分镜单元，捆绑首帧/尾帧/视频/音频/提示词
2. **双端异构**：桌面端三栏工作台追求效率，移动端卡片流追求可用性
3. **Git 是唯一的真理源**：策略、工作流、Shot Card 模板、审核人权限全部版本化
4. **渐进式填充**：节点异步完成，Shot Card 动态组装，最小可审集合解锁审核
5. **审计与执行物理隔离**：治理层故障不影响 OpenClaw 已启动任务
6. **不可变审计链**：审计记录通过 Merkle Root 定期锚定到 Git
7. **音频并行前置**：音频生成节点与视觉节点并行启动，审核时同步决策

---

此架构下，OpenClaw 保持"高灵活快速迭代"特性，治理层通过 Shot Card 捆绑审核 + GitOps 版本控制 + 桌面/移动双端网关 + AI 预留窗口，为短剧管线提供可审计、可回溯、可渐进自动化的质量闸门。
